Three gateways, one contract

LiteLLM, Bifrost, and Helicone all claim to serve the same four tiers — tier-fast, tier-heavy, tier-frontier, tier-private — behind one client-facing contract. None of the three honor every clause of it today. Where they diverge is specific and load-bearing: not vibes, but exact config fields, source paths, and one shared blind spot (fallback that looks configured but never fires on a real error).

The health-monitor circuit breaker (discover.monitor.health) does mark a provider unhealthy after a real failure — but under path-pinned /router/<name> addressing, an unhealthy pinned provider just fails outright rather than silently substituting a pool member.
Untested fix path: Helicone's Unified API (https://ai-gateway.helicone.ai) takes a comma-separated, provider-qualified fallback chain directly in the model field (e.g. tier-fast/ollama,tier-heavy/ollama) — architecturally distinct from /router/<name>. Unconfirmed whether it works against self-hosted Ollama with tier-alias names, not just Helicone-hosted/BYOK keys.

Bottom line

LiteLLM's fallback ladder is the only one actually wired end-to-end — treat it as the reference behavior, not a baseline every gateway already matches. Bifrost and Helicone each have a real, separate, still file-based fallback mechanism that was never exercised in this repo's config: Bifrost's retries-and-fallbacks feature, and Helicone's Unified API. Neither requires standing up Postgres/SQLite/Cloud mode — the fix in both cases is switching mechanisms, not adding infrastructure. Nothing in the verified evidence says otherwise; claims that a database is required were explicitly checked and rejected.
